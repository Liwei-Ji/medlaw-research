# 學習筆記

這份筆記記錄本專案在概念與工程上實際碰到、學到的東西。操作說明請看各層 README;這裡談的是「為什麼」與「學到什麼」。

## 檢索系統,不是 RAG

RAG 是「檢索 + 用 LLM 生成答案」。本專案只做檢索、沒有生成:問題經查詢解析與索引比對後,答案由查表與模板組出,全程不呼叫語言模型。這樣做是為了可離線、零成本、結果可稽核。用常見詞彙說,它是一套資訊檢索(IR)系統,外觀做成對話介面;與 RAG 的差別在於少了「生成」那一層。

## 用到的概念

### 1. 資料工程 / 爬蟲
- 爬有狀態的 ASP.NET 政府網站(`__VIEWSTATE`、session、POST 查詢)。
- 破解「結果清單僅開放前 500 筆分頁」:把裁判日期區間**遞迴對半切**到每段 ≤500 筆再合併。
- 自雜亂 HTML 解析結構資料、逐案下載全文,再以規則擷取條號 / 刑度 / 緩刑 / 法官等欄位。

### 2. 資訊檢索(IR)核心
- **倒排索引(inverted index)**:搜尋引擎的根本資料結構,本專案自行實作。
- **中文斷詞**:CJK 沒有詞界,採 **bigram / trigram**(以字元 n-gram 取代分詞)。
- **兩種檢索範式**:結構化 / 分面檢索(法規·條號·法院·年度·類別篩選) vs **全文檢索**。
- **排序與呈現**:詞頻 / bm25 排序、snippet 摘要、命中高亮。

### 3. 查詢理解(NLU-lite)
把自然語言問句拆成「結構化篩選 + 自由關鍵字」(例如「甲基安非他命 114年 刑事」→ 內文詞 + 年度 + 類別),讓「搜尋」用起來像「問答」。

### 4. 預先計算(precomputation)
統計數字、案由旗標在**建置期**算好、烤進資料,執行期只查表——把運算搬到離線並快取的思維。

### 5. 系統工程取捨:全文 DB 的體積之旅
全文檢索原想用 sql.js(SQLite 編成 WASM)的 FTS5 trigram,過程學到幾件事:
- sql.js 的**預建版不含 FTS5**(`no such module: fts5`),不是所有 SQLite 建置都一樣。
- 對全 14,031 筆建 trigram 索引,DB 膨脹到 **755MB**;瀏覽器根本載不動。
- 放棄 SQLite、改自製純 JS 引擎後,又發現「載入後在瀏覽器建 bigram 索引」對全量約需 30–40 秒、太慢。
- 最終方案:**只帶全文 body(gzip 後 78MB)、`DecompressionStream` 解壓後直接用 `indexOf` 掃描**(每次查詢約數十毫秒、免索引)——無 WASM、涵蓋全 14,031 筆;代價是首次下載 78MB、記憶體約 400MB。
- 版控上也有取捨:App 執行期抓的 78MB gz 必須是「一般 commit 的檔」(**GitHub Pages 不服務 Git LFS**);而 262MB 的原始語料壓成 `.jsonl.gz`(80MB,仍 <100MB)也用一般 commit 保存,連 Git LFS 都免了。

一路上也理解到這些環境限制會**反過來決定設計**:CSP 擋外部資源、`file://` 不能 `fetch`、GitHub 單檔 100MB 上限。

### 6. 產品 / 資訊設計
對話式 UI、漸進揭露(載入更多、縮小範圍 chip)、生成感動畫、來源引用、交叉統計熱力圖(資料視覺化)。
